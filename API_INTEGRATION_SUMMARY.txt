═══════════════════════════════════════════════════════════════════════════════
                         ERROR HANDLING PATTERNS
═══════════════════════════════════════════════════════════════════════════════

All API methods follow this pattern:

try {
  // Check if Supabase is configured (warn if not)
  // Execute Supabase query
  // Check for errors and wrap in ApiError
  // Return data
} catch (err) {
  // Log detailed error information
  // Re-throw with context
}

Components catch errors at:
- ComponentLevel (try-catch in useEffect)
- Fallback to default/local data
- Log errors to console for debugging

═══════════════════════════════════════════════════════════════════════════════
                            ERROR LOGGING FORMAT
═══════════════════════════════════════════════════════════════════════════════

All errors logged with:
- err.name          : Error type (e.g., "ApiError")
- err.message       : Human readable message
- err.code          : Supabase error code
- err.status        : HTTP status (if applicable)
- err.details       : Additional context

Example console output:
{
  name: "ApiError",
  message: "Failed to fetch products",
  code: "PGRST116",
  status: 400,
  details: { /* Supabase error details */ }
}
